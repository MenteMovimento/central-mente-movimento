3. Testes locais

Como testar antes de publicar

  1. Abrir a pasta do projeto no editor.
  2. Iniciar um servidor local na pasta do projeto, por exemplo python -m http.server 4177.
  3. Abrir http://127.0.0.1:4177 no browser.
  4. Testar login, pesquisa, criar/editar sócio, pagar quota, exportar e gestor de administradores.
  5. Validar o JavaScript com node --check app.js.
  6. Confirmar que não existem erros visíveis na consola do browser.

Nunca testar alterações diretamente em produção sem antes confirmar localmente que o fluxo principal continua a funcionar.

4. Publicação

Como atualizar o site

  1. Alterar os ficheiros necessários.
  2. Testar localmente.
  3. Fazer commit no Git com uma mensagem clara.
  4. Enviar para o GitHub no branch main.
  5. A Vercel faz o deploy automaticamente.
  6. Abrir o website publicado e confirmar a alteração.

Fluxo esperado: computador local → GitHub → Vercel → website público.

5. Supabase

Base de dados, SQL e RLS

O código do site não altera sozinho a estrutura da base de dados. Quando há um campo novo, também é preciso executar SQL no Supabase.

  1. Entrar no Supabase.
  2. Abrir o projeto da associação.
  3. Ir a SQL Editor.
  4. Colar o script da pasta supabase/.
  5. Clicar em Run.
  6. Testar no site publicado.
Script Quando usar
schema.sql Criação inicial da base de dados.
harden-rls.sql Reforço das permissões e Row Level Security.
add-quota-paid-at.sql Adiciona a data em que a quota foi paga.
add-approval-minute-number.sql Adiciona o Nº de Ata de Aprovação.

6. APIs

Funções seguras da Vercel

Algumas ações não devem ser feitas diretamente no browser porque precisam de permissões fortes. Por isso existem funções na pasta api/.

api/create-user.js

Cria novos utilizadores autorizados através do Gestor de Administradores.

api/delete-user.js

Remove utilizadores com validações para impedir ações inseguras.

A variável SUPABASE_SERVICE_ROLE_KEY fica apenas na Vercel, em Environment Variables. Nunca deve ir para GitHub, config.js ou qualquer ficheiro público.

7. Segurança

Cuidados obrigatórios de cibersegurança

  • Manter Row Level Security ativa no Supabase.
  • Dar acesso administrativo apenas a quem realmente precisa.
  • Ativar 2FA/MFA nas contas GitHub, Vercel e Supabase.
  • Guardar passwords num local seguro e controlado pela associação.
  • Fazer backups/exportações regulares dos dados e testar recuperação.
  • Não colocar chaves privadas no código público.
  • Rever o histórico de alterações quando houver dúvidas.
  • Remover acessos quando uma pessoa sai da associação.

Regra prática: cada pessoa deve ter a sua própria conta. Evitar contas partilhadas; quando forem inevitáveis, guardar a password num gestor seguro e mudar a password sempre que alguém com acesso sair da associação.

8. Rotina semestral

Plano semestral de cibersegurança

De 6 em 6 meses deve existir uma revisão técnica formal, registada por escrito com data, responsável e resultado.

Tarefa O que fazer Registo mínimo
Backups Exportar dados dos sócios e confirmar que existe cópia protegida fora da conta principal. Data, responsável, local da cópia e ficheiro exportado.
Teste de recuperação Confirmar que o ficheiro exportado abre corretamente e que os dados essenciais estão presentes. Resultado do teste e erros encontrados.
Passwords administrativas Rodar passwords das contas institucionais/admin e chaves partilhadas, evitando variações previsíveis. Contas revistas e confirmação de atualização.
Acessos Rever GitHub, Vercel, Supabase e Gestor de Administradores da app. Lista de pessoas que ficaram com acesso.
MFA/2FA Confirmar que administradores têm autenticação de dois fatores ativa. Serviços verificados e contas sem 2FA.
Histórico e logs Rever alterações recentes, eliminações e ações administrativas suspeitas. Resumo da revisão e ações tomadas.

Além da revisão semestral, mudar passwords imediatamente se houver suspeita de acesso indevido, perda de computador/telemóvel, saída de alguém com acesso, ou partilha acidental de credenciais.

8.1 Backups

Política de backups

Periodicidade mínima

Backup semestral obrigatório. Se houver muitas alterações de sócios/quotas, fazer também backup mensal.

O que guardar

Exportação da lista de sócios, registo de quotas, utilizadores autorizados e scripts SQL importantes.

Onde guardar

Duas cópias: uma em armazenamento institucional protegido e outra em local separado, com acesso restrito.

Proteção

Evitar enviar backups por email comum. Se possível, guardar em pasta protegida e com controlo de acessos.

Um backup só é útil se puder ser recuperado. Pelo menos uma vez por semestre, abrir o ficheiro exportado e confirmar que contém nomes, contactos, quotas e campos principais.

8.2 Passwords

Política de passwords e contas

  • Usar passwords únicas para GitHub, Vercel, Supabase, email institucional e app.
  • Usar frases-password longas ou passwords geradas por gestor de passwords.
  • Ativar 2FA/MFA em todas as contas administrativas.
  • Fazer revisão semestral das passwords administrativas e institucionais.
  • Mudar passwords partilhadas sempre que alguém com acesso deixa a associação.
  • Não reutilizar passwords antigas nem criar variações óbvias, como mudar apenas o ano ou um número.
  • Não guardar passwords em documentos públicos, chats, folhas de cálculo abertas ou emails.
  • Desativar contas que já não são usadas.

Nota técnica: recomendações modernas de cibersegurança dão prioridade a passwords únicas, longas, MFA e mudança imediata quando há suspeita de compromisso. A revisão semestral deve evitar criar passwords fracas ou previsíveis.

8.3 Incidentes

O que fazer se houver suspeita de problema

  1. Suspender ou remover o acesso da conta suspeita no Gestor de Administradores.
  2. Mudar passwords das contas afetadas e confirmar 2FA/MFA.
  3. Rodar a SUPABASE_SERVICE_ROLE_KEY se houver suspeita de exposição da chave.
  4. Rever histórico da app, utilizadores autorizados e logs do Supabase/Vercel.
  5. Fazer backup do estado atual antes de alterações profundas.
  6. Confirmar se houve acesso a dados pessoais dos sócios.
  7. Informar a direção e avaliar obrigações RGPD com apoio adequado.
  8. Registar o incidente: data, pessoa que detetou, impacto, ações tomadas e conclusão.

Nunca apagar logs ou histórico para “limpar” o problema. Primeiro preservar evidências, depois corrigir o acesso.

9. Alterações futuras

Como adicionar um campo novo

  1. Criar o SQL na pasta supabase/.
  2. Executar o SQL no Supabase.
  3. Adicionar o campo ao formulário em index.html.
  4. Atualizar leitura, gravação, validação e exportação em app.js.
  5. Atualizar textos de português/inglês se o campo aparecer no interface.
  6. Atualizar styles.css se o layout precisar.
  7. Atualizar manuais e relatórios.
  8. Testar localmente e publicar pelo GitHub.

10. Continuidade

Como a associação deixa de depender de uma pessoa

GitHub

Transferir o repositório para uma organização GitHub da associação.

Vercel

Transferir o projeto para uma equipa Vercel controlada pela associação.

Supabase

Garantir que a organização/projeto Supabase pertence à associação.

Acessos

Ter pelo menos duas pessoas com acesso administrativo e 2FA ativo.

O ideal é usar emails institucionais da associação para propriedade, faturação, recuperação de conta e alertas técnicos.

11. Checklist técnica

Antes de publicar qualquer alteração

  • O pedido está claro e limitado ao necessário?
  • Os ficheiros alterados são os corretos?
  • O site foi testado localmente?
  • node --check app.js passou sem erros?
  • Se houve campo novo, o SQL foi criado e executado?
  • As permissões continuam corretas para admin, operador e consulta?
  • O commit foi enviado para o GitHub?
  • A Vercel publicou e o site final foi confirmado?
